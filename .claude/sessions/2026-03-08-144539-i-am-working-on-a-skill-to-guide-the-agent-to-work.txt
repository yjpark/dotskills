  ---
  What the skill should teach the agent to do

  Goal: produce a clean, logical change graph вҖ” not one big blob of everything.

  Principles:
  1. Never mix concerns in one change вҖ” if you're fixing a bug while adding a
  feature, split them
  2. Describe changes early вҖ” jj describe -m "..." before moving on
  3. Working copy is always safe вҖ” you can't lose work, so be bold about jj new
  4. Navigate freely вҖ” use jj edit to return to any change and amend it
  retroactively

  The workflow loop:
  jj new -m "what I'm about to do"   # start a named, empty change
  # ... make changes ...
  jj st                               # verify what changed
  jj new -m "next thing"              # move on, previous work is preserved

  When interrupted / context switching:
  # Current work is automatically saved (it's already a commit)
  jj new -m "wip: quick fix"         # start separate change for the
  interruption
  # ... do the fix ...
  jj edit <original-change-id>       # return to previous work

  Cleanup before "done":
  jj log                             # review the change graph
  jj squash                          # merge trivial fixups into parent
  jj split                           # split if a change grew too large
  jj describe -m "proper message"    # rewrite descriptions
  jj abandon                         # drop a change that's no longer needed

  ---
  Scope questions to decide before writing

  1. Should the skill include jj git push / sync with git remotes? Or stay
  purely local-history focused?
  2. Should it cover jj rebase? (reordering changes) вҖ” useful but adds
  complexity
  3. Target: always a linear chain of changes, or allow branching change graph?
  4. Should it instruct the agent to describe changes before or after writing 
  code? (before is better for intent clarity)

  My recommendation for a first version: linear chain only, no remote sync, 
  descriptions before coding. That's the smallest thing that gives a clean
  history.

  What's your reaction to this framing? Any of these feel off, or is there a
  specific workflow pattern you had in mind?

  Final skill design: jujutsu-basics

  Scope

  In scope: Linear chain of revisions, bookmarks, squash workflow, planning
  Out of scope: jj split, jj edit, jj rebase, jj git push, remote sync, conflict
   resolution, issue tracker integration

  Commands used

  jj new, jj new -m, jj describe -m, jj bookmark create -r @, jj bookmark set -r
   @, jj squash, jj log, jj status

  Bookmark naming

  <slug> (kebab-case). Issue tracker prefix reserved for future.

  Revision description format

  One-line summary

  - Refs: <task context>
  - What: <planned changes>
  - Checks: <completion conditions>

  Workflow

  Phase 1 вҖ” Planning (interactive, blocking)

  1. Receive task
  2. Plan revisions вҖ” each coherent, ordered by dependency, with clear checks
  3. Discuss with user, clarify, suggest improvements
  4. Do NOT proceed until user approves

  Phase 2 вҖ” Execution

  For each planned revision:

  1. jj new -m "<description>" вҖ” if this is the first revision AND there are
  multiple planned revisions, append the full plan after a --- separator
  2. jj bookmark create -r @ <slug> (first revision) or jj bookmark set -r @ 
  <slug> (subsequent)
  3. Notify user: bookmark created/moved

  Then the working loop:

  1. jj new вҖ” create unnamed staging area
  2. Make file changes in small steps
  3. When changes clearly advance the parent revision вҶ’ jj squash
  4. Notify user on each squash with brief summary
  5. Repeat until parent revision's checks all pass

  Move to next planned revision. Repeat.

  Phase 3 вҖ” Completion

  1. Leave the final empty staging revision in place (user may want final
  touches)
  2. Verify bookmark points to last completed revision
  3. Generate a summary for the user: what was done, how many revisions,
  bookmark location

  Conflict handling: Stop immediately, alert user with context, wait for manual
  resolution.
